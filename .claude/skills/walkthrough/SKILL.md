# Walkthrough

전달받은 `<도메인>/<기능>` (예: `routine-todo/categories`) 기능의 구현 내용을 **비개발 동료도 따라갈 수 있는 HTML walkthrough**로 만든다. 코드를 바꾸지 않는다 — 설명 문서만 생성한다. 흰 테마, 자체 완결형(브라우저에서 바로 열림)

## 0. 입력 확인

인자는 `<도메인>/<기능>` 형태다 (예: `routine-todo/categories`). 다음을 읽는다:

- plan 파일: `docs/work/<도메인>/<기능>.md` — 결정값·스텝·위험영역·미해결의 정본. **없으면 멈추고** 사용자에게 `/plan`·`/run-plan`을 먼저 돌렸는지 확인한다.
- 실제 구현: 그 기능이 만든/바꾼 소스. `git diff`, `git log`, 또는 해당 도메인 패키지(`domain/.../<도메인>`, `user-api/.../<기능>`)와 테스트를 읽어 plan과 실제가 맞는지 대조한다.
- 스타일/골격 템플릿: `docs/work/_walkthrough-template.html`.

plan과 실제 코드가 어긋나면 **실제 코드가 우선**이고, 어긋난 점은 문서의 "후속" 섹션에 짧게 적는다.

## 1. 생성 규칙

`docs/work/_walkthrough-template.html`를 복제해 내용을 채운다.

- `<style>` 블록과 클래스는 **그대로 둔다** (룩앤필 단일 소스). 색·폰트를 새로 만들지 않는다.
- 템플릿 안내 주석(`채우는 법 …`)과 `{{플레이스홀더}}`는 산출물에서 **모두 제거**한다.
- 섹션은 템플릿의 1~10 골격을 따른다. **이 기능에 해당 없는 섹션은 통째로 삭제**한다(데이터모델/위험영역/후속이 없으면 뺀다). 억지로 채우지 않는다.
- 내용은 **코드 재진술이 아니라 "무엇을·왜"** 중심. 흐름과 의사결정을 설명한다.
  - 핵심 설계 결정 = plan의 "결정값(확정)" 항목을 근거와 함께.
  - 위험영역 = plan에 `(위험)` 표시가 있을 때만. 트랜잭션 경계·정합성·함정을 `.danger`/`.note` 박스로.
  - 테스트 구성 = 테스트 클래스별로 무엇을 검증하는지 + 검증 결과(verifier/reviewer 통과 여부 알면 함께).
- 산문은 평서/존대(`~합니다`/`~다`). 코드 식별자(클래스·필드·경로)는 원문 유지하고 `<code>`로 감싼다.
- 짧은 코드 인용은 `<pre>`로. 길게 붙여넣지 말고 핵심만.

## 2. 출력

- 저장 위치: `docs/work/<도메인>/<기능>-walkthrough.html`
- 생성 후 사용자에게 경로와 `open <경로>` 명령을 알리고, 어떤 섹션을 담았는지/뺐는지 한 줄로 요약한다.
- 코드·커밋·PR은 하지 않는다.

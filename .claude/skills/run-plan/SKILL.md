# Run Plan

전달받은 **이슈 번호 또는 제목**(예: `/run-plan 21`, `/run-plan 내 방 슬롯 조회`)에 해당하는 GitHub 이슈를 입력으로, 본문의 남은 `- [ ]` 스텝을 **자율 반복으로 burn-down**한다. 이슈가 정본이다. 이 스킬은 추측하지 않는다 — 계획에 없는 결정이 필요하면 멈춘다.

## 0. 이슈 찾기 + 진입 가드 (반드시 먼저)

- 인자가 숫자면 `gh issue view <번호> --json number,title,body,state`로 읽는다.
- 인자가 제목/문구면 `gh issue list --state open --search "<문구>" --json number,title`로 찾는다. 0건이면 못 찾았다고 알리고 멈춘다. 2건 이상이면 후보를 보여주고 어느 번호인지 사용자에게 확인한다.

읽은 이슈에서 아래 중 하나라도 걸리면 **구현하지 말고 멈춰서 사용자에게 알린다**:

- 이슈가 없다/못 찾았다 → `/plan`을 먼저 돌리라고 안내
- 이슈가 이미 closed → 끝난 작업. 종료.
- 본문에 미확정/미해결 표시가 남아 있다 → 무엇이 미확정인지 알리고 `/plan`으로 확정하라고 안내
- 남은 `- [ ]` 스텝이 없다 → 이미 완료. 종료.

가드를 통과하면 구현을 시작한다.

## 1. 자율 루프 (한 번에 한 스텝)

남은 `- [ ]` 스텝이 없어질 때까지 아래를 반복한다. 한 iteration = 스텝 1개.

1. 본문 체크리스트에서 맨 위의 안 닫힌 `- [ ]` 스텝을 고른다.
2. 스텝을 구현한다. docs/claude/backend.md "산출물을 어느 모듈에 두는가"를 따른다:
   - Entity·Repository·migration → `domain` 모듈
   - Service·Controller·DTO → `user-api`(admin 기능이면 `admin-api`)
   - 에러는 `common.error.ErrorResponse` 형식 / `BusinessException` + `ErrorCode`
   - 여러 테이블 쓰기는 `@Transactional`, 조회 서비스는 `@Transactional(readOnly = true)`
   - migration 버전은 `domain/src/main/resources/db/migration/`의 최신 다음 번호. **이미 적용된 migration 파일은 수정하지 않고 새 번호만 추가**한다.
   - **이슈 범위 안에서만** 작업한다. 본문에 없는 파일·스텝을 임의로 추가하지 않는다. 빠진 게 보이면 멈추고 보고한다(→ "멈춤 조건").
3. `./gradlew test` 를 돌린다. 깨지면 고친다 — **단 최대 3회까지만**. 3회 안에 못 통과하면 멈추고 원인을 보고한다(→ "멈춤 조건").
   - **기존 테스트를 삭제·약화(assertion 제거 등)해서 통과시키지 않는다.** 테스트가 옳고 코드가 틀린 게 기본 가정이다.
4. 통과하면 이슈 본문의 그 스텝을 `- [x]`로 갱신한다: `gh issue view <번호> --json body -q .body`로 현재 본문을 받아 해당 줄만 `[ ]`→`[x]`로 바꾼 뒤 `gh issue edit <번호> --body-file <임시파일>`로 덮어쓴다. (GitHub가 task-list 진행률을 자동 표시한다.)
5. 다음 iteration으로.

스텝 도중 **결정·구현 방향이 바뀌었으면**(예: plan에 적힌 방식 대신 다른 방식 채택) 본문의 결정값을 최신으로 고치고, **왜 바꿨는지는 `gh issue comment <번호>`로 남긴다.** 본문 = 최신 정본, 댓글 = 변경 이력.

검증(verifier·reviewer)은 스텝마다 하지 않고 **"마무리"에서 한 번** 돌린다.

## 2. 멈춤 조건 (추측 금지)

다음 상황에서는 루프를 멈추고 사용자에게 상황을 보고한다 — 임의로 진행하지 않는다. 멈출 때 진행상황을 `gh issue comment`로 남긴다:

- 계획에 없던 결정값이 필요해졌다 (enum·보상액·타임존 등) → 미해결로 취급, 멈춤
- 다른 도메인 계약을 확정해야 한다 → dependency로 남기고 멈춤
- 본문에 없는 파일·스텝이 필요해졌다 → 범위 밖, 멈추고 보고
- 테스트를 3회 고쳐도 계속 깨진다 → 무한 수정 대신 멈추고 원인 보고
- 마무리 검증(`verifier` 미달 / `reviewer` blocking)을 5회 보완해도 통과 못 하거나, 보완에 plan에 없는 결정이 필요하다 → 멈추고 보고

## 3. 이어하기

스텝이 많아 한 번에 다 못 끝내거나 멈춤 조건으로 중단됐으면, 진행상태가 이슈 본문의 `- [x]`에 남아 있으므로 `/run-plan <이슈번호>` 를 다시 호출하면 안 닫힌 스텝부터 이어간다.

## 4. 마무리

- 이번 run에서 `user-api`(또는 `admin-api`) 컨트롤러/DTO를 새로 추가·변경했으면, 그 파일들에 `/add-swagger`를 호출해 Swagger 애노테이션을 붙인다. 컨벤션 정본은 `add-swagger` — run-plan은 위임만 하고 규칙을 재서술하지 않는다.
- 멈춤이든 완료든, 누적 변경에 대해 검증한다:
  - `verifier` — **항상**. plan 완료기준 대비 완료성·테스트 충실도.
  - `reviewer` — **위험영역(재화·완료·뽑기·인증·집 미션·migration)을 건드렸을 때만** 깊게. 정합성·보안·인가.
- 미달·blocking 지적이 나오면 **fix → 재검증** 루프를 돈다:
  - 이슈 범위 안에서 고칠 수 있는 것(빠진 구현, 약한/빈 테스트, 정합성 버그)은 고치고 해당 검증을 다시 돌린다. **최대 5회**.
  - 5회 안에 통과 못 하거나, 보완에 **plan에 없는 결정이 필요**하면 → 멈춤 조건으로 보고(추측 금지).
- 일반 코드 품질·버그 리뷰가 필요하면 `/code-review`를 직접 돌리라고 사용자에게 안내한다.
- **plan 확정 내용을 spec에 반영** → `sync-spec <이슈번호>` 를 호출한다. 이슈 본문을 근거로 spec 정본(`../rougether-spec`)을 최신화한다. 자세한 규칙은 `sync-spec` skill 참고.
- `bootRun` 등 long-running 세션을 남기지 않는다.
- **이슈를 직접 close하지 않는다.** close는 **PR 머지 시 자동**으로 한다 — 구현이 끝나면 사용자에게 "`/pr`로 PR을 만들 때 본문에 `closes #<이슈번호>` 를 넣어라(머지되면 이슈 자동 close)"고 안내한다.
- **커밋·PR은 하지 않는다.

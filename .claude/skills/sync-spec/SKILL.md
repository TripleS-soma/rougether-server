# Sync Spec

전달받은 `<도메인>/<기능>` (예: `routine-todo/categories`)의 plan 파일을 근거로, `/plan`에서 사람과 확정하며 구체화·수정한 도메인 계약을 spec 정본(`../rougether-spec`)에 반영한다. **plan 파일이 정본 근거다 — 구현 코드를 다시 대조하지 않는다.** spec working tree만 고치고 커밋·PR은 하지 않는다.

## 0. 입력 확인

인자는 `<도메인>/<기능>` 형태다. 다음을 확인한다:

- plan 파일: `docs/work/<도메인>/<기능>.md` — 확정값·변경사항·해소된 미해결의 근거. **없으면 멈추고** 사용자에게 `/plan`을 먼저 돌렸는지 확인한다.
- spec repo: `../rougether-spec`가 있는지 확인한다. 없으면 멈추고 보고한다.
- plan에 "미해결"이 남아 있으면, 그 항목은 spec에 쓰지 않는다(아직 확정 안 됨). 확정된 것만 대상.

## 1. 반영 규칙

**기존 spec과 달라진 점 + 주요하게 알아야 할 사항만** 반영한다. 근거는 plan 파일이다. plan에 있는 모든 걸 옮기지 않는다 — 필드 단위 상세·구현 내부는 spec에 쓰지 않는다(그건 서버 repo `docs/work/`의 몫).

- 대상은 plan이 건드린 영역의 spec 문서다:
  - `domains/<도메인>/` — 해당 도메인의 기능·데이터·API 계약
  - 관련된 `api.md` · `erd.md` · `features.md` · `open-questions.md`
- 반영하는 것:
  - **새로 추가된 엔드포인트** → spec `api.md` 표에 행으로 추가한다(method·path·목적·요청/응답 핵심).
  - **요청/응답 필드 변경** → 필드가 추가·삭제·이름변경되면 해당 표에 반영한다(예: 응답에 `visibility` 추가, `aiReviewStatus` 삭제).
  - **그 외 기존 spec과 어긋나거나 바뀐 계약** (path 변경, 상태 전이 방식 변경 등).
  - **계약상 주요 결정** (enum 허용값, 보상 수치, 타임존 같은 정책 기준).
- 반영하지 않는 것: DTO 필드 전체 나열, 서비스/리포지토리 내부, 테스트, 구현 단계 세부.
- 해소된 open-question은 `open-questions.md`에서 닫고(또는 "결정됨"으로 표시), 결정 내용을 해당 도메인 문서에 반영한다.
- spec의 표기 관례를 따른다 — 기존 문서의 문체·섹션 구조·표 형식을 그대로 이어쓴다. 새 포맷을 만들지 않는다.
- **날짜·"확정됨(YYYY-MM-DD)" 같은 시점 표기를 spec 문서에 쓰지 않는다.** 계약 내용만 적는다.

## 2. 멈춤 조건 (추측 금지)

- plan 파일에 근거가 없는 변경 → 임의로 spec에 쓰지 않는다. 멈추고 사용자에게 확인한다.
- plan의 결정과 현재 spec이 정면충돌하는데 어느 쪽이 최신인지 plan에 근거가 없다 → 멈추고 사용자에게 확인한다.
- 다른 담당자 도메인 문서를 크게 바꿔야 한다 → 멋대로 확정하지 말고 보고한다(CLAUDE.md: 다른 담당 도메인 계약 임의 확정 금지).

## 3. 마무리

- **spec repo는 커밋·PR을 하지 않는다.** working tree에만 반영한다 — 커밋/PR은 사용자가 spec repo에서 직접 한다(CLAUDE.md: 사용자 확인 전 기획 문서 PR 금지).
- 무엇을 어느 파일에 반영했는지 **변경 요약**을 사용자에게 보고한다.
- 반영할 게 없으면 "spec과 일치 — 반영할 변경 없음"으로 보고한다.

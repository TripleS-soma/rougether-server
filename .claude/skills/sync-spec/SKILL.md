# Sync Spec

구현하며 확정·변경된 도메인 계약을 spec 정본(`../rougether-spec`)에 반영한다. **근거는 두 가지 중 하나다 — GitHub 이슈 본문(`/plan`이 만든 정본) 또는 코드 변경(diff).** 어느 쪽이든 계약 변화만 추려 spec에 옮긴다. 근거에 없는 변경은 임의로 쓰지 않는다.

`/plan`은 계획을 **이슈**로 남기고 `docs/work/` 같은 plan 파일을 만들지 않는다. plan 파일을 찾지 않는다.

## 0. 근거 정하기 (입력 확인)

인자와 상황을 보고 근거를 정한다:

- **이슈 근거** — 인자가 이슈 번호(예: `21`)거나, `/run-plan` 마무리에서 넘어온 경우. `gh issue view <번호> --json title,body,state`로 본문을 읽는다. 확정된 계약의 근거다. 인자가 제목/문구면 `gh issue list --state all --search "<문구>" --json number,title`로 찾고, 0건이면 코드 근거로 넘어가고 2건 이상이면 사용자에게 어느 번호인지 확인한다.
- **코드 근거** — 이슈가 없고 직접 코드만 바꾼 경우(단독 실행). 현재 브랜치가 `main`과 다른 점 + 커밋 안 된 작업 변경에서 계약 변화를 파악한다:
  - `git diff main...HEAD` — 브랜치에 쌓인 커밋
  - `git status` · `git diff` — 아직 커밋 안 된 변경
  - 인자로 `<도메인>` 또는 `<도메인>/<기능>`을 받으면 그 범위로 좁혀 본다(선택).

공통 확인:
- spec repo `../rougether-spec`가 있는지 확인한다. 없으면 멈추고 보고한다.
- 이슈 본문에 **미확정/미해결 표시가 남아 있으면**, 그 항목은 spec에 쓰지 않는다(아직 확정 안 됨). 확정된 것만 대상.

## 1. 근거에서 계약 변화 추리기

근거(이슈 본문 또는 diff)에서 **externally-observable 계약이 바뀐 것만** 추린다. spec에 옮길 후보:

- **새로 추가·변경된 엔드포인트** — 코드 근거면 컨트롤러의 `@GetMapping`/`@PostMapping` 등 method·path 변화. 이슈 근거면 본문 헤더의 `<METHOD> /api/v1/...`.
- **요청/응답 필드 변경** — DTO 필드가 추가·삭제·이름변경됨(예: 응답에 `visibility` 추가, `aiReviewStatus` 삭제).
- **동작 규칙 변경** — 필터·정렬·소유권 판정·상태 전이·데이터 소싱 규칙 등 계약상 관측 가능한 동작이 바뀜(예: today 투두를 overdue 누적에서 정확 마감일 매칭으로).
- **계약상 주요 결정** — enum 허용값, 보상 수치, 타임존 같은 정책 기준.

추리지 않는 것(코드 근거일 때 특히): 리포지토리/서비스 내부 구현, private 헬퍼, 테스트, DTO 필드 전체 나열. 관측 가능한 계약을 바꾸지 않는 리팩토링은 spec에 안 쓴다.

## 2. spec에 반영

**기존 spec과 달라진 점 + 주요하게 알아야 할 사항만** 반영한다. 근거에 있는 모든 걸 옮기지 않는다.

- 대상 문서:
  - `domains/<도메인>/` — 해당 도메인의 기능·데이터·API 계약
  - 관련된 `api.md` · `erd.md` · `features.md` · `open-questions.md`
- 새 엔드포인트 → spec `api.md` 표에 행 추가(method·path·목적·요청/응답 핵심).
- 필드·동작 변경 → 해당 표·서술을 고친다. 기존 spec이 옛 계약을 서술하고 있으면 그 문장을 정정한다.
- 해소된 open-question은 `open-questions.md`에서 닫고(또는 "결정됨"으로 표시), 결정 내용을 도메인 문서에 반영한다.
- spec의 표기 관례를 따른다 — 기존 문서의 문체·섹션 구조·표 형식을 그대로 이어쓴다. 새 포맷을 만들지 않는다.
- **날짜·"확정됨(YYYY-MM-DD)" 같은 시점 표기를 spec 문서에 쓰지 않는다.** 계약 내용만 적는다.

## 3. 멈춤 조건 (추측 금지)

- 근거(이슈 본문·diff)로 의도를 확정할 수 없는 변경 → 임의로 spec에 쓰지 않는다. 멈추고 사용자에게 확인한다. (코드 근거일 때: 이 diff가 의도된 계약 변경인지 부수적 변경인지 애매하면 묻는다.)
- 근거의 결정과 현재 spec이 정면충돌하는데 어느 쪽이 최신인지 근거로 판단이 안 된다 → 멈추고 사용자에게 확인한다.
- 다른 담당자 도메인 문서를 크게 바꿔야 한다 → 멋대로 확정하지 말고 보고한다(CLAUDE.md: 다른 담당 도메인 계약 임의 확정 금지).

## 4. 마무리

- **완료 후 커밋·PR을 한다.**
- 무엇을 어느 파일에 반영했는지, 그리고 어떤 근거(이슈 #번호 / 코드 diff)를 썼는지 **변경 요약**을 사용자에게 보고한다.
- 반영할 게 없으면 "spec과 일치 — 반영할 변경 없음"으로 보고한다.
